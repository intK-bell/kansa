# 写真アップロード時の写真名・初回コメント 要件メモ

更新日: 2026-04-04

## 目的

写真アップロード時に、各写真ごとに以下を同時に保存できるようにする。

- 写真名
- 初回コメント

初回コメントは専用フィールドではなく、通常コメント1件として保存する。
`description` のような別フィールドは追加しない。

## 入力要件

- 写真名は必須
- 写真名の最大文字数は 20 文字
- 初回コメントは任意
- 初回コメントの最大文字数は 50 文字

## UI要件

- 複数ファイル選択時は、選択したファイル一覧を表示する
- 各ファイル行に `写真名` と `初回コメント` の入力欄を表示する
- 写真名の初期値は元ファイル名を入れる
- 初回コメントの初期値は空欄にする
- 一括アップロード前に、各ファイルごとの内容を確認・修正できるようにする
- アップロード中はアップロードボタンを無効化する
- アップロード中はアップロードボタンに進行中と分かるアニメーションを付ける

## 補助操作要件

- `初回コメントを全件に反映` ボタンを用意する
- `写真名を連番で反映` ボタンを用意する
- 写真名の連番反映は、1行目の入力欄の現在値をベースにする
- 反映形式は `ベース名_001`, `ベース名_002`, `ベース名_003` のように 3 桁ゼロ埋め昇順とする
- 連番の開始番号は `001` とする
- 写真名の連番反映は、既存入力がある場合でも上書き対象とする
- 初回コメントの全件反映は、既存入力がある場合でも上書き対象とする
- 写真名の連番反映を実行する前に、上書き確認ダイアログを表示する
- 初回コメントの全件反映を実行する前に、上書き確認ダイアログを表示する

## 実装手順

1. 要件を固定する  
   写真アップロード時に、各ファイルごとに `写真名` は必須、`初回コメント` は任意にする。  
   保存形式は `fileName` と、通常コメント1件の自動作成で統一する。

2. フロントの入力UIを追加する  
   アップロード対象ファイル一覧の各行に `写真名` 入力欄と `初回コメント` 入力欄を出す。  
   現状の `uploadFiles()` は `FileList` をそのまま順に送るだけなので、送信前にファイルごとの入力値を保持する状態管理が必要。  
   あわせて、`初回コメントを全件に反映` と `写真名を連番で反映` の補助ボタンを配置する。

3. アップロード前バリデーションを入れる  
   `写真名` が空ならそのファイルは送信しない。  
   `写真名` は 20 文字以内、`初回コメント` は 50 文字以内で制御する。  
   空文字の `初回コメント` は未入力扱いにする。  
   補助ボタンによる一括反映時は、上書き確認ダイアログを必須にする。

4. 写真保存APIのリクエストを拡張する  
   `POST /folders/{folderId}/photos` の body に、既存の `photoId` `originalS3Key` `previewS3Key` `fileName` に加えて `initialComment` を渡す。

5. backend で `initialComment` を受ける  
   `finalizePhoto()` に `initialComment` の trim とバリデーションを追加する。  
   空文字は `null` 扱いにし、値がある場合のみコメント作成処理へ進む。

6. 写真レコード保存後に通常コメントを1件作る  
   既存のコメント保存形式に合わせて、`PK=PHOTO#{photoId}` `SK=COMMENT#{timestamp}#{commentId}` のアイテムを作成する。  
   フィールド構造は既存の `createComment()` に合わせる。

7. 写真のコメント要約フィールドも更新する  
   `latestCommentAt` `latestCommentBy` `latestCommentByName` `updatedAt` を、初回コメント作成時にも更新する。  
   これにより未読表示や一覧表示の整合を保つ。

8. 監査ログを2本出す  
   既存の `photo.create` はそのまま残す。  
   `initialComment` がある場合だけ、初回コメント分として `comment.create` 相当の監査ログも追加で出す。  
   これにより「写真作成」と「初回コメント作成」を監査上で分けて確認できる。

9. 既存UIとの整合を取る  
   写真一覧は既に `photo.fileName` を表示しているため、新規入力された写真名がそのまま見える想定。  
   初回コメントは通常コメントとして保存されるため、既存の表示・編集・削除・PPT出力にそのまま乗せる。

10. 動作確認をする  
   写真名が一覧に表示されること。  
   初回コメントがコメント欄に1件入ること。  
   PPT にそのコメントが出ること。  
   監査ログで `photo.create` と `comment.create` の両方が確認できることを確認する。  
   アップロード中はボタンが無効化され、進行中アニメーションが表示されることを確認する。

## 補足メモ

- 写真名は既存の `fileName` を流用する
- 初回コメントは通常コメントとして保存する
- `description` のような別フィールドは追加しない
- 複数ファイル同時アップロード時は、各ファイルごとに個別入力できるUIが必要
- 写真名の連番は `1行目の現在入力値` を基準にして `_<3桁連番>` を付ける
- 写真名連番反映・初回コメント全件反映の両方で、上書き前確認ダイアログを出す
- アップロード完了時は、画面操作を止めるモーダルダイアログより、成功件数や失敗件数が分かる完了メッセージ表示を優先する

- 選択済みファイルと入力内容をまとめて破棄する `キャンセル` ボタンを追加する

### rev1時点の要件

#### 入力要件

- 写真名の最大文字数は 20 文字
- 初回コメントは任意
- 初回コメントの最大文字数は 50 文字

#### UI要件

- 複数ファイル選択時は、選択したファイル一覧を表示する

- 選択済みファイルと入力内容を破棄する `キャンセル` ボタンを表示する

#### 補助操作要件

- `1つ目のコメントを全件に反映` ボタンを用意する
- `1つ目の写真名を連番で反映` ボタンを用意する
- 写真名の連番反映は、1行目の入力欄の現在値をベースにする
- コメント全件反映は、1行目の初回コメント入力欄の現在値をベースにする
- 反映形式は `ベース名_001`, `ベース名_002`, `ベース名_003` のように 3 桁ゼロ埋め昇順とする

#### 保存要件

- 写真名は既存の `fileName` を流用して保存する
- 初回コメントは通常コメント1件として保存する
- `description` のような別フィールドは追加しない
- 初回コメントの投稿者表示は通常コメントと同じ扱いにする

#### 完了通知要件

- アップロード完了時はモーダルダイアログではなく完了メッセージを表示する
- 完了メッセージには成功件数、必要に応じて失敗件数や重複件数を含める

### rev1の画面操作フロー

1. ユーザーがフォルダを選び、写真ファイルを複数選択する
2. 選択直後に、アップロード対象一覧を画面に表示する
3. 一覧の各行に、元ファイル名、写真名入力欄、初回コメント入力欄を表示する
4. 写真名には元ファイル名を初期表示する
5. 初回コメントは空欄で開始する
6. 補助操作として以下を表示する
7. `1つ目のコメントを全件に反映`
8. `1つ目の写真名を連番で反映`
9. `キャンセル`
10. `1つ目のコメントを全件に反映` 実行時は確認ダイアログを出す
11. 確認後、1行目の初回コメント入力欄の現在値を全行へ上書き反映する
12. `1つ目の写真名を連番で反映` 実行時は確認ダイアログを出す
13. 確認後、1行目の写真名入力欄の現在値をベースに `ベース名_001` `ベース名_002` ... の形式で全行へ上書き反映する
14. `キャンセル` 実行時は確認ダイアログを出す
15. 確認後、選択済みファイルと入力済み内容を破棄する
16. ユーザーは必要に応じて各行を個別修正する
17. `アップロード` 実行時に入力値を検証する
18. 正常時は写真保存を行い、初回コメントがあれば通常コメント1件として同時保存する
19. 保存後は完了メッセージを表示する

### rev1の注意メモ

- ボタン名はUI文言としてそのまま使えるように固定する
- コメント反映元は共通入力欄ではなく1行目のコメント欄に変更する
- キャンセル時は、ファイル選択 input の値だけでなく、画面上の入力一覧も確実に消す
- フォルダ切替時やアップロード完了時の下書きクリア仕様と矛盾しないようにする

## rev2

更新日: 2026-04-04

### rev2での修正点

- アップロード対象が 1 枚だけの時は、`1つ目のコメントを全件に反映` と `1つ目の写真名を連番で反映` は非表示にする
- 各 `upload-draft-row` ごとに、右上の個別削除ボタン `❌` を追加する
- 一括キャンセルだけでなく、「5枚選んだうち1枚だけ外す」ような個別除外をできるようにする

### rev2時点の追加・修正要件

#### UI表示制御

- アップロード対象が 2 枚以上の時だけ、以下の補助ボタンを表示する
- `1つ目のコメントを全件に反映`
- `1つ目の写真名を連番で反映`
- アップロード対象が 1 枚の時は、上記 2 ボタンは非表示にする
- `キャンセル` ボタンは 1 枚の時も表示してよい

#### 個別キャンセル要件

- 各 `upload-draft-row` の右上に `❌` ボタンを表示する
- `❌` ボタンを押すと、その行の写真だけアップロード対象一覧から除外する
- 個別除外時は、対象ファイルに対応する写真名・初回コメント入力内容も一緒に破棄する
- 個別除外後は、残り行数に応じて補助ボタンの表示状態を再判定する
- 個別除外後、対象が 0 件になった場合は、一覧全体を閉じた状態に戻す

### rev2の画面操作フロー差分

1. 写真を複数選択して一覧表示したあと、各行の右上に `❌` を表示する
2. ユーザーは不要な写真だけ `❌` で個別除外できる
3. 一覧が 1 枚だけになった時点で、`1つ目のコメントを全件に反映` と `1つ目の写真名を連番で反映` は非表示になる
4. 一覧が 0 件になった場合は、選択状態と入力一覧を閉じる

### rev2の注意メモ

- 個別除外後に行番号表示がある場合は、詰めて再採番する
- `❌` は削除対象が明確に見える位置として各行右上に固定する
- 一括キャンセルと個別キャンセルの役割を分ける
  - 一括キャンセル: 選択済み全件を破棄
  - 個別キャンセル: 対象行だけ除外

## rev4

更新日: 2026-04-04

### rev4での追加点

- 各 `upload-draft-row` に、該当写真の軽量サムネイル表示を追加する

### rev4時点の表示要件

#### サムネイル表示

- `upload-draft-head` の下に、その行に対応する写真のサムネイルを表示する
- サムネイルは「内容確認の補助」が目的なので、ものすごく軽量な表示でよい
- 元画像をそのまま大きく描画するのではなく、小さめのプレビューとして表示する
- サムネイルは各行の写真名・初回コメント入力欄と対応関係が一目で分かる位置に置く
- サムネイル表示はアップロード前のローカル確認用途であり、保存データ項目は増やさない

#### 表示方針

- 画像サイズは小さめ固定でよい
- 比率は崩さず、枠内に収める表示を優先する
- 表示が重くならないように、一覧確認に支障が出ない程度の軽さを優先する
- 写真が読み込めない場合でも、入力欄の操作自体は継続できるようにする

### rev4の画面イメージ方針

1. 行タイトルと右上の `❌`
2. その直下に小さなサムネイル
3. その下に `写真名` と `初回コメント` 入力欄

### rev4の注意メモ

- サムネイルは装飾ではなく、対象写真の取り違え防止が目的
- 複数枚アップロード時でもスクロール負荷が増えすぎないようにする
- 軽量表示を優先し、必要以上に大きいプレビューにはしない

## rev5

更新日: 2026-04-04

### rev5での方針変更

- `HEIC/HEIF` はそのまま扱わず、アップロード前に `JPG` へ変換する
- サムネイルだけでなく、アップロード本体も `JPG` に統一する
- 保存先の画像形式も `JPG` を前提にする

### rev5時点の画像変換要件

#### 変換方針

- `HEIC/HEIF` を選択した場合は、アップロード前にクライアント側で `JPG` に変換する
- 変換後の `JPG` をそのままアップロード対象とする
- サムネイル表示も、変換後の `JPG` を使う
- `PNG/JPG/JPEG` など、もともとブラウザで扱いやすい画像はそのまま扱ってよい

#### 保存方針

- backend へ送る画像本体は、最終的に `JPG` として扱う
- ファイル形式の統一を優先し、後続処理や表示の互換性を安定させる
- `HEIC` のまま保存する運用は行わない

### rev5時点のUI要件

- 変換が必要な画像では、アップロード開始前に変換処理が入ることを前提とする
- 変換中は、ユーザーに進行中であることが分かる表示を出す
- 変換中もアップロード中と同様に、ボタンの多重操作は防ぐ

### rev5時点の性能・体感要件

#### 体感上の影響

- `HEIC/HEIF -> JPG` 変換のぶん、アップロード開始まで少し待ち時間が増える
- 特に複数枚同時アップロード時は、端末によって少し重く感じる可能性がある
- ただし、運用上は互換性向上のメリットを優先する

#### 性能配慮

- 変換は端末負荷が急増しないよう、基本的に順次処理を優先する
- 画質とサイズのバランスを取り、必要以上に重い `JPG` を生成しない
- 高解像度画像をそのまま巨大な `JPG` にしないように配慮する

### rev5の意図

- iPhone 利用を前提にしても、`HEIC` のブラウザ差分に振り回されないようにする
- サムネイル表示、一覧表示、PowerPoint 出力の安定性を上げる
- 利用者にとって「画像が見える/見えない」の端末差を減らす

### rev5の注意メモ

- `HEIC` 特有の情報は `JPG` 化で失われる可能性がある
- `HEIC` より `JPG` の方がサイズが増える場合がある
- そのため、変換品質や縮小方針は実装時に慎重に決める

### rev5の扱い

- rev5 の `HEIC/HEIF -> JPG` 変換方針は採用しない
- 理由は、実装時に `ERR_LIBHEIF format not supported` が発生し、対象HEICを安定変換できなかったため
- 現時点では rev4 方針へ戻し、`HEIC` は端末・ブラウザがネイティブに表示できる範囲でサムネイル表示する
- つまり、変換は行わず、軽量サムネイル表示までを採用状態とする
