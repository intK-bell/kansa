# 料金プラン差別化 実装計画

更新日: 2026-04-04

対象要件: [pricing-requirements.md](/Users/aokikensaku/Documents/Devapps/kansa/pricing-requirements.md)

方針:
- いきなり全面改修せず、`判定ロジック` → `管理画面` → `LP` → `PPT出力` → `アーカイブ処理` の順で段階実装する
- 既存のサブスク基盤を活かし、プラン差分だけを追加する
- 無料/有料の差分は、表示だけでなく `作成制限` `切り替え制限` `出力差分` `保存差分` まで一貫させる

## フェーズ1: 課金・制限制御の土台（Backend）

- [ ] プラン差別化ルールを `backend/src/billing.js` に集約する
  - 無料 `512MB`
  - 有料 `1GB / 5GB / 10GB`
  - フリー時フォルダ上限 `2`
  - フリー時保存期間 `30日`
- [ ] フリープラン判定用の共通関数を用意する
  - `フリーかどうか`
  - `フォルダ作成可能か`
  - `フリーへ戻せるか`
  - `PPT透かし対象か`
- [ ] フリーへ戻す条件判定を `backend/src/api.js` の `action=free` に追加する
  - `使用容量が512MB未満`
  - `フォルダ数が2個以下`
  - `使用容量` は `アーカイブ済みデータを含める`
- [ ] `action=free` のガードは UI 依存にせず、API 単体で必ず拒否できるようにする
  - 条件未達なら `billingMode` / `currentPlan` を更新しない
  - 条件未達内容をレスポンスに含める
  - 直接 API を叩かれても無料へ戻せないようにする
- [ ] 条件未達時に、UIがそのまま出せるレスポンス形式を決める
  - 固定文言
  - 容量未達
  - フォルダ数未達
  - 両方未達
- [ ] フォルダ作成APIに、フリープラン時の `2個上限` ガードを追加する
- [ ] 監査ログに、フリー切り替え成功 / 条件未達を残すか整理し、必要なら追加する

完了条件:
- API 単体で、無料へ戻せる / 戻せない の判定が安定して返る
- フリープラン中は3個目のフォルダを作れない

## 先に確認しておくリスク

- `action=free` は現状ほぼ無条件で通せる実装に見えるため、先に Backend ガードを入れないと UI だけ直しても抜ける
- アーカイブは `非表示` と `容量計算に含む` の両立が必要なため、一覧APIと集計APIの責務を分けて考える必要がある
- PPT 透かしは `写真の上に並べて表示` に寄せて実装を簡素化するが、出力実装箇所の調査は早めにやる方が安全
