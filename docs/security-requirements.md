# セキュリティ要件整理

更新日: 2026-04-04

## 目的

- 本サービスのローンチ前に、最低限満たすべきセキュリティ要件を明確化する
- 認証、認可、写真アップロード、課金、監査、運用の各観点で、実装要件と運用要件を整理する
- リリース判定で使える `ローンチ必須` と、リリース後も継続対応する `改善推奨` を分ける

## 対象範囲

- Frontend: `frontend/`
- Backend API: `backend/src/api.js`
- Stripe Webhook: `backend/src/stripe-webhook.js`
- IaC: `backend/template.yaml`
- 運用手順: `docs/production-ops.md`

## 守る対象

- Cognito による利用者アカウント
- ID トークン、OAuth 状態値、招待トークン
- 写真原本、プレビュー画像、PowerPoint 出力ファイル
- フォルダ情報、コメント、監査ログ、課金メタデータ
- Stripe の Subscription / Checkout / Webhook 連携情報
- AWS リソース設定、環境変数、秘密情報

## 脅威前提

- 未認証アクセス
- 認可不備による他チームデータ閲覧・更新
- XSS によるトークン窃取
- 不正アップロード、危険ファイル混入、過大ファイル投入
- S3 / API / Webhook の設定不備による外部公開
- Stripe webhook なりすまし、重複処理、課金不整合
- 操作履歴不足による追跡不能
- 秘密情報の漏えい、誤設定、権限過大

## セキュリティ方針

- 認証は Cognito を正とし、未認証アクセスは許可しない
- 認可は `チーム` `フォルダ` `作成者` 単位で最小権限にする
- 保存データと転送経路は、公開範囲を最小化する
- 課金、削除、権限変更など重要操作は監査可能にする
- 本番設定は allowlist と秘密情報分離を前提にする
- 実装だけでなく、運用手順と監視をセットでローンチ条件にする

## 要件

### 1. 認証

- `ローンチ必須` API は `Authorization: Bearer <Cognito ID Token>` を必須とする
- `ローンチ必須` Cognito User Pool は MFA を有効化し、パスワードポリシーを維持する
- `ローンチ必須` OAuth リダイレクト URL とログアウト URL は、本番ドメインとローカル開発用だけに限定する
- `ローンチ必須` トークン失効、期限切れ、改ざん時は `401` または `403` を返し、曖昧な成功扱いをしない
- `改善推奨` フロントの ID トークン保存先は `localStorage` 依存を減らし、将来的に `sessionStorage` または BFF / Cookie 化を検討する

### 2. 認可

- `ローンチ必須` すべての API で、対象データがログインユーザーの所属チームに属することを検証する
- `ローンチ必須` 管理者専用操作は、メンバー操作と明確に分け、サーバー側で権限判定する
- `ローンチ必須` 写真・コメントの編集削除は、作成者本人または管理者だけに制限する
- `ローンチ必須` フォルダ単位パスワードは平文保存しない
- `ローンチ必須` フォルダパスワード必須操作では、管理者を除きサーバー側で検証を実施する
- `改善推奨` 認可失敗理由はユーザーに必要最小限だけ返し、内部状態の詳細は返さない

### 3. Web / フロントエンド

- `ローンチ必須` 本番フロントは HTTPS のみで公開する
- `ローンチ必須` Content Security Policy を設定し、少なくとも script-src を厳格化する
- `ローンチ必須` ユーザー入力値は DOM 挿入時に HTML として解釈させない
- `ローンチ必須` トークン、フォルダパスワード、Webhook secret、Stripe secret を画面・ログ・URL クエリへ出さない
- `ローンチ必須` `frontend/config.js` に秘密情報を置かない
- `改善推奨` `Referrer-Policy`, `X-Content-Type-Options`, `Permissions-Policy` を明示設定する

### 4. API / CORS

- `ローンチ必須` API の CORS 許可 Origin は、本番フロント URL とローカル開発 URL に限定する
- `ローンチ必須` レスポンスヘッダと API Gateway 側の CORS 設定を整合させ、ワイルドカード許可を残さない
- `ローンチ必須` API Gateway のスロットリング設定を有効にし、変更時は運用手順に反映する
- `ローンチ必須` 独自ドメイン切替後は、動作確認が終わり次第 `execute-api` 直アクセスを無効化する
- `改善推奨` CloudFront + WAF を前段に置き、IP レート制限と一般的な攻撃シグネチャ防御を追加する

### 5. 写真アップロード / S3

- `ローンチ必須` 写真保存バケット、PPT 出力バケットともに Public Access Block を有効化する
- `ローンチ必須` S3 バケットポリシーで匿名公開を許可しない
- `ローンチ必須` S3 CORS は必要な Origin / Method / Header のみに絞る
- `ローンチ必須` アップロード用署名 URL の有効期限は短時間に限定し、再利用前提にしない
- `ローンチ必須` アップロード時に許可 MIME タイプを画像系に限定する
- `ローンチ必須` ファイルサイズ上限を定め、フロントとバックエンドの両方で制御する
- `ローンチ必須` `finalizePhoto` 時に、対象オブジェクトが想定バケット・想定キー配下に存在することを確認する
- `改善推奨` マルウェアスキャンまたは画像デコード検証を導入し、非画像ファイル混入を下げる
- `改善推奨` S3 の暗号化、ライフサイクル、アクセスログを明示設定する

### 6. データ保護

- `ローンチ必須` すべての保存データは AWS の標準暗号化を有効にする
- `ローンチ必須` DynamoDB / S3 のデータ削除系処理は、対象スコープをサーバー側で再確認する
- `ローンチ必須` 個人情報に相当する項目は、必要最小限だけ保持する
- `ローンチ必須` 退会・部屋削除時は、削除対象範囲を明確にし、実装と運用手順を一致させる
- `改善推奨` バックアップ / リストア方針と保持期間を別紙で定義する

### 7. Stripe / 課金連携

- `ローンチ必須` `STRIPE_SECRET_KEY` と `STRIPE_WEBHOOK_SECRET` はリポジトリに置かず、安全な設定経路から注入する
- `ローンチ必須` Webhook は署名検証に失敗したら即拒否する
- `ローンチ必須` Webhook は冪等に処理し、同じイベントやセッションを二重反映しない
- `ローンチ必須` 部屋削除時は Stripe 停止の成否を確認し、失敗時はアプリ内削除へ進まない
- `ローンチ必須` 課金状態変更は監査ログとアプリ内メタデータに残す
- `改善推奨` Stripe API 障害時の再試行戦略と手動復旧手順を運用手順へ追記する

### 8. 監査ログ / 監視

- `ローンチ必須` 以下の重要操作は `kind=audit` で記録する
- 認証関連エラー
- 権限変更
- フォルダ作成 / 削除 / パスワード更新
- 写真作成 / 更新 / 削除 / 一括削除
- コメント作成 / 更新 / 削除
- PowerPoint 出力
- 課金開始 / 変更 / 解約 / Webhook 反映
- `ローンチ必須` 監査ログには `requestId`, `actor`, `action`, `target`, `result`, `reason` を含める
- `ローンチ必須` トークン、パスワード、Webhook secret、Stripe secret はログに出さない
- `ローンチ必須` CloudWatch Alarm の通知先を設定し、受信確認まで完了させる
- `ローンチ必須` ログ保存期間を設定し、運用で参照できるようにする
- `改善推奨` 4XX / 5XX / throttle 傾向の定期レビューを行う

### 9. 秘密情報 / 設定管理

- `ローンチ必須` 本番秘密情報は `.env` やソースコード直書きで管理しない
- `ローンチ必須` 本番環境変数の設定元、更新者、更新手順を明文化する
- `ローンチ必須` 秘密情報のローテーション手順を用意する
- `ローンチ必須` 権限付き IAM は最小権限を原則とし、不要な `*` 権限を避ける
- `改善推奨` Secrets Manager または SSM Parameter Store に統一する

### 10. 運用 / インシデント対応

- `ローンチ必須` 本番反映は IaC と手順書に基づいて実施し、コンソール手修正を常態化させない
- `ローンチ必須` 障害時の一次切り分け手順を `API`, `S3`, `Cognito`, `Stripe` の各系統で用意する
- `ローンチ必須` インシデント時の連絡先、一次対応者、復旧判断者を決める
- `ローンチ必須` 本番変更後は、最低限の受け入れ確認を実施する
- `改善推奨` 依存ライブラリの脆弱性確認を定期実施する
- `改善推奨` `npm audit fix --force` が必要な依存更新は破壊的変更として扱い、未使用依存は削除を優先する
- `改善推奨` 依存更新後は `npm audit` が0件であることと、Backend の構文確認が通ることを記録する

## ローンチ必須チェックリスト

- Cognito MFA と本番 Callback / Logout URL の確認完了
- API 認可が全エンドポイントで有効
- 重要操作の監査ログ出力確認完了
- S3 バケット公開設定の遮断確認完了
- S3 CORS が allowlist 化済み
- API CORS が allowlist 化済み
- 本番 HTTPS 化完了
- `execute-api` endpoint 無効化完了
- Stripe secret / webhook secret の安全な設定完了
- Webhook 署名検証、冪等処理、課金停止フロー確認完了
- CloudWatch Alarm 通知受信確認完了
- ログ保存期間設定完了
- インシデント時の運用手順確認完了

## 現状確認メモ（2026-04-04）

- Cognito MFA とパスワードポリシーは `backend/template.yaml` に設定済み
- API Gateway のデフォルトスロットリングは `backend/template.yaml` に設定済み
- 監査ログ運用は `docs/production-ops.md` に記載あり
- フロントは `frontend/main.js` で ID トークンを `localStorage` に保持している
- 写真バケットの CORS は `backend/template.yaml` で `AllowedOrigins: ["*"]` になっている
- API レスポンスヘッダは `backend/src/api.js` と `backend/src/stripe-webhook.js` で `access-control-allow-origin: *` を返している
- `ExportBucket` は明示的な Public Access Block 設定が見当たらないため、確認と固定化が必要
- 本番フロント設定は `frontend/config.js` で配布されるため、秘密情報を置かない前提を維持する必要がある

## ローンチ前の優先対応

### 優先度A

- API と S3 の CORS を allowlist 化する
- Export 用バケットの公開遮断設定を明示する
- CSP を導入し、XSS から `localStorage` 上の ID トークンを守る
- `execute-api` 直アクセスを無効化する

### 優先度B

- アップロード MIME 制限、サイズ上限、拡張子制御を仕様化して実装へ反映する
- 秘密情報の保管場所とローテーション手順を確定する
- WAF 導入可否を決める

### 優先度C

- トークン保管方式の見直し
- S3 アクセスログ、マルウェア対策、脆弱性定期点検の運用化

## 非対象

- 法務文面そのものの作成
- 個人情報保護方針の文言確定
- 顧客向けセキュリティ説明資料の整備
